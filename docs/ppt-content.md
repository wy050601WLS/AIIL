## 第 8 页 — 技术讲解：SSE 流式推送

**标题**：技术详解 — SSE 流式推送

**什么是 SSE？大白话**：

普通请求就像寄快递 — 你下单后，快递全部打包好才送到。SSE 就像流水线 — 东西做好一件就送一件，你不用等全部做好。

**为什么不用 WebSocket？大白话**：

WebSocket 是双向对讲机，双方都能说话。SSE 是广播，只有服务器能说。AI 对话只需要服务器往浏览器推内容，不需要浏览器往服务器推，所以 SSE 更简单。

**实现流程**：

```
1. 前端用 fetch 发 POST 请求到 /chat
2. 后端收到请求，用 httpx 异步调用 AI 接口（stream=True）
3. AI 每生成一个字，后端立刻通过 SSE 推给前端
4. 前端用 ReadableStream 逐字接收，追加到消息里
5. AI 回答完毕，后端关闭连接，把完整回复存到数据库
```

**后端代码**：
```python
async with client.stream("POST", url, json=payload) as resp:
    async for line in resp.aiter_lines():
        yield f"data: {content}\n\n"
```

**大白话**：这行代码的意思是 — 打开一个「流」连接，AI 每吐出一行文字，我就立刻用 `yield` 丢给前端。`yield` 就像「接力棒传递」，收到一个字就传一个字，不等后面的内容。

**前端代码**：
```javascript
const reader = response.body.getReader()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  onToken(parsedText)
}
```

**大白话**：这段代码的意思是 — 打开一个「水龙头」，不停地读取后端推过来的文字。每读到一个字，就调用 `onToken` 显示到页面上。`done` 是 true 表示 AI 回答完了，关掉水龙头。

**效果**：首字延迟从几秒降到 AI 首 chunk 时间，用户立刻看到 AI 在「打字」

## 第 9 页 — 技术讲解：Markdown 渲染防抖

**标题**：技术详解 — Markdown 渲染与防抖

**问题大白话**：

AI 回复里可能有标题、列表、代码块。前端要把这些 Markdown 格式转成网页能显示的 HTML。但 AI 打字很快，每秒几十个字，如果每个字都重新转一次，页面就像闪屏一样疯狂闪烁。

**什么是防抖？大白话**：

防抖就像电梯门 — 有人进来，门就重新计时。只有没人进来了，门才关。代码里：AI 每输出一个字，计时器就重置。只有 150ms 内没有新字了，才触发渲染。

```
AI 输出第 1 个字 → 不渲染，等 150ms
AI 输出第 2 个字 → 重置计时器，再等 150ms
AI 输出第 3 个字 → 重置计时器，再等 150ms
150ms 内没有新字 → 触发渲染
AI 停止回答 → 立刻渲染最终版本
```

**代码**：
```javascript
watch(() => props.content, (val) => {
  if (renderTimer) clearTimeout(renderTimer)
  if (!props.loading) {
    renderedContent.value = DOMPurify.sanitize(marked.parse(val))
    return
  }
  renderTimer = setTimeout(() => {
    renderedContent.value = DOMPurify.sanitize(marked.parse(val))
  }, 150)
})
```

**大白话**：
- `watch` — 监听 AI 回复内容的变化
- `clearTimeout(renderTimer)` — 有新字来了，取消之前的渲染计划
- `!props.loading` — AI 不在输出了（回答完了），立刻渲染
- `setTimeout(..., 150)` — AI 还在输出，等 150ms 再渲染
- `DOMPurify.sanitize(marked.parse(val))` — 把 Markdown 转成 HTML，再过滤掉可能的恶意代码

**渲染管线大白话**：

```
AI 原始文字 → marked 把 Markdown 转成 HTML
           → highlight.js 给代码块加颜色
           → DOMPurify 过滤掉危险的 HTML（防止 XSS 攻击）
           → 显示到页面
```

**效果**：CPU 占用降低 90%，页面不卡不闪

## 第 11 页 — 技术讲解：无限滚动

**标题**：技术详解 — 无限滚动

**问题大白话**：一个对话可能聊了几百条消息，全部一次加载，页面要等很久。

**解决方案大白话**：

就像翻微信聊天记录 — 先显示最近的消息，你往上翻才加载更早的。这样打开对话很快，不用等所有消息都加载完。

```
第一次打开对话 → 只加载最近 50 条消息
用户往上滚动 → 检测到滚到顶部 → 加载更早的 50 条
加载后 → 补偿滚动位置，页面不跳动
```

**滚动位置补偿 — 核心难点大白话**：

想象你在看一本竖着的长图。你正在看中间某一段，突然上面又加了一段新内容，你看到的位置就往下移了，你得自己往回滚。

解决方法：加载前记住「页面有多高」，加载后算「新页面比旧页面高了多少」，把滚动条调回去。

```javascript
const 旧高度 = el.scrollHeight          // 记住加载前页面高度
await chatStore.loadMoreMessages()      // 加载旧消息（插到列表最前面）
el.scrollTop = el.scrollHeight - 旧高度  // 把滚动条调回原来看到的位置
```

**大白话**：
- `scrollHeight` — 整个页面的总高度
- `scrollTop` — 当前滚动条的位置
- 加载旧消息后页面变高了，用「新高度 - 旧高度」就是新增的高度，把滚动条往下调这么多，你就还是看到原来的位置

**AI 上下文管理大白话**：

发给 AI 的历史消息不能太多，否则超出 AI 的「记忆容量」（token 限制）。所以：
- 最多发最近 50 条消息
- 最近 10 条保留图片（AI 能看图），更早的只保留文字（省 token）

## 第 13 页 — 技术讲解：AI 智能搜索

**标题**：技术详解 — AI 智能搜索

**传统搜索 vs AI 搜索大白话**：

传统搜索就像在图书馆用电脑查书名 — 你必须输入准确的关键词，差一个字都找不到。

AI 搜索就像问图书管理员 — 你可以用大白话问「有没有适合初学者的 Python 书？」，管理员能理解你的意思，推荐合适的书。

| 方式 | 输入 | 结果 |
|------|------|------|
| 传统搜索 | 关键词匹配 | 精确但不智能 |
| AI 搜索 | 自然语言提问 | 理解语义，智能推荐 |

**实现流程大白话**：

```
用户问「我有哪些编程资料？」
  → 后端把用户的所有资料整理成一份清单
  → 把清单和用户的问题一起发给 AI
  → AI 看完清单后，挑出相关的资料，告诉用户推荐理由
  → 前端把 AI 的回答显示出来
```

**为什么用异步？大白话**：

同步就像你去餐厅点菜，厨师给你做菜的时候，你只能站在那里等，不能干别的。异步就像你点了菜拿了号，可以先去逛超市，菜好了手机通知你。

后端调 AI 需要几秒钟，如果用同步，这几秒钟其他用户的请求全部排队等着。改成异步后，调 AI 的时候后端可以同时处理其他请求。

## 第 16 页 — 技术讲解：注册登录与安全

**标题**：技术详解 — 认证与安全

**JWT 认证流程大白话**：

```
注册 → 你设的密码被加密成乱码存起来（原始密码谁都看不到）
登录 → 输入密码，后端跟你存的乱码对比，对了就发一张「通行证」
通行证（Token）→ 前端保存在浏览器里
每次请求 → 自动把通行证带上，后端一看就知道你是谁
通行证过期（24小时）→ 自动跳回登录页，重新登录
```

**bcrypt 加密大白话**：

bcrypt 就像一台「碎纸机」— 你把密码塞进去，出来的是一堆乱码。这台碎纸机有个特点：你可以拿一个新的密码和那堆乱码对比，看是不是同一个密码做的。但你没法从乱码反推出原始密码。

```python
# 加密：密码 → 乱码
bcrypt.hashpw("123456".encode(), bcrypt.gensalt())
# 结果：$2b$12$XyZ...（一串谁都看不懂的乱码）

# 验证：拿用户输入的密码和数据库存的乱码对比
bcrypt.checkpw("123456".encode(), stored_hash)  # 返回 True 或 False
```

**安全防护大白话**：

| 风险 | 大白话 | 防护措施 |
|------|--------|----------|
| 密码泄露 | 数据库被偷了，密码不能明文存 | bcrypt 加密 |
| 接口被刷 | 有人疯狂发请求，服务器扛不住 | 限流，每分钟最多 60 次 |
| AI 回复注入 | AI 回复里可能夹带恶意代码 | DOMPurify 过滤 |
| 越权操作 | 用户 A 偷看用户 B 的数据 | 每个接口检查是不是本人 |
| SQL 注入 | 攻击者通过输入框往数据库注入恶意代码 | ORM 参数化查询 |

## 第 17 页 — 技术讲解：前端性能优化

**标题**：技术详解 — 前端性能优化

**问题**：初始打包 2MB，加载慢

**优化 1 — Element Plus 按需导入大白话**：

之前：把 Element Plus 的所有组件（几百个）全部打包，但其实只用了一部分。
之后：用插件自动检测你用了哪些组件，只打包用到的。就像搬家只搬需要的东西，不搬整个仓库。

**优化 2 — highlight.js 语言裁剪大白话**：

highlight.js 支持 190 多种编程语言的代码高亮，但你的用户不可能用到所有语言。只保留常用的 15 种（JavaScript、Python、Java 等），其他的砍掉。就像手机只装需要的 APP，不装应用商店里的所有 APP。

**优化 3 — 代码分包大白话**：

```javascript
manualChunks(id) {
  if (id.includes('node_modules/vue'))     return 'vendor-vue'
  if (id.includes('node_modules/marked'))  return 'vendor-markdown'
  if (id.includes('node_modules/highlight.js')) return 'vendor-hljs'
}
```

这段代码的意思是：把不同功能的代码分成不同的文件。Vue 的代码放一个包，Markdown 的放一个包，代码高亮的放一个包。浏览器可以同时下载多个小文件，比下载一个大文件快。

**结果**：2MB → 666KB，缩小 66%

## 第 18 页 — 技术讲解：后端异步与数据库优化

**标题**：技术详解 — 后端异步与数据库优化

**异步改造大白话**：

```python
# 之前：同步
def ask_resources():
    response = requests.post(url, json=payload)  # 卡住！
    return response.json()

# 之后：异步
async def ask_resources():
    async with httpx.AsyncClient() as client:
        response = await client.post(url, json=payload)  # 不卡！
        return response.json()
```

**大白话**：
- `def` 是普通函数，执行的时候会「卡住」，后面的代码得等它完成
- `async def` 是异步函数，执行到 `await` 的时候会「让出」CPU，去处理别的请求
- `requests.post` 是同步请求，发出去就傻等
- `httpx.AsyncClient` 是异步请求，发出去不傻等，AI 回来了再继续

就像餐厅：同步是一个厨师一次只做一道菜，做完才做下一道。异步是一个厨师同时看好几个锅，哪个锅好了就处理哪个。

**数据库优化大白话**：

| 优化项 | 大白话 |
|--------|--------|
| 复合索引 | 给数据库加「目录」，查东西更快 |
| 列表限制 200 条 | 不让一次返回太多数据，就像快递不让你一次寄 1000 件 |
| AI 上下文 50 条 | 不给 AI 喂太多历史消息，超出它的「记忆容量」会出错 |

## 第 19 页 — 项目难点与解决

**标题**：项目难点与解决

| 难点 | 大白话描述 | 解决方案 |
|------|-----------|---------|
| AI 回复慢 | 用户发了问题，盯着空白页面好几秒 | 用 SSE 流式推送，AI 边回答边显示 |
| 页面闪烁 | AI 打字太快，每个字都重新渲染，页面疯狂闪 | 加 150ms 防抖，像电梯门一样等人进完再关 |
| 长对话卡顿 | 一个对话几百条消息，全部加载很慢 | 只加载最近 50 条，往上翻才加载更多 |
| 丢回复 | 重新生成失败，原来的回复也没了 | 先把旧回复存起来，失败了自动恢复 |
| 查询变慢 | 数据多了，列表查询越来越慢 | 加索引（目录）+ 限制返回数量 |
| 搜索阻塞 | 调 AI 的时候，其他用户的请求全排队 | 改成异步，调 AI 的同时能处理别的请求 |

## 制作建议

**配色**：深蓝 + 白底，科技感

**字体**：标题用黑体/思源黑体，正文用微软雅黑

**每页原则**：
- 文字不超过 6 行
- 多用表格和图示，少用大段文字
- 关键数字放大加粗
- 技术页配代码截图或流程图

**动画**：
- 要点逐条出现
- 流程图逐步展开
- 不要花哨转场

**技术讲解页建议**：
- 左侧放流程图/架构图
- 右侧放关键代码 + 大白话解释
- 底部放效果对比（优化前 vs 优化后）
